# NewRAG 后端方案架构（详细规划）

版本：v1.0  
日期：2026-01-05  
范围：仅后端方案与接口占位（LLM 通过外部 API 接入）

---

## 1. 目标与非目标

### 目标
- 🎯 构建“结构解析 → 分层检索 → 证据融合 → 生成 → 校验”的可验证后端方案。
- 🎯 提供稳定、可扩展、可替换的索引与 LLM 网关接口。
- 🎯 为前端提供清晰 API（问答、证据、结构树、反馈、索引管理）。
- 🎯 定义验收基线，确保可测、可追溯。

### 非目标
- 不做前端实现与视觉设计。
- 不内置具体大模型推理（仅预留 LLM API 接口）。
- 不覆盖多模态（图像/表格识别）深度实现，仅留扩展点。

---

## 2. 核心假设与约束

- 文档主要为中文长文档（PDF/DOCX/MD/TXT）。
- 文档存在一定层级结构（章/节/段落/列表/表格），但允许结构不完整。
- 需要支持混合检索（稀疏 + 稠密）。
- LLM 通过外部 API 接入（本地或云端均可），本方案只保留接口。
- 评测与测试用例需保证单次后台测试 < 60s。

---

## 3. 状态定义（可验证）

### 3.1 文档状态
- `RAW`：原始文件已入库。
- `PARSED`：结构解析完成，生成结构树与原始段落。
- `CHUNKED`：完成层次化切分，生成 chunk。
- `EMBEDDED`：chunk 向量生成完成。
- `INDEXED`：稀疏/稠密索引同步完成。
- `READY`：可检索状态。

### 3.2 查询状态
- `RETRIEVE_COARSE`：章节级粗召回。
- `RERANK_FINE`：段落级精排。
- `FUSE_EVIDENCE`：证据去冗余与冲突消解。
- `GENERATE_ANSWER`：调用 LLM 生成答案。
- `VERIFY_ANSWER`：反向检索/一致性校验。
- `DONE`：结果就绪并输出。

### 3.3 任务状态
- `PENDING` / `RUNNING` / `SUCCESS` / `FAILED`（用于解析、索引构建等后台任务）

## 4. 流程描述（端到端）

### 4.1 文档入库与索引流程
1. 上传文档 → 校验格式 → 存储原文（RAW）。
2. 结构解析：提取章节/标题/段落/表格（PARSED）。
3. 层次化切分：生成带路径的 chunk（CHUNKED）。
4. 向量化：为 chunk 生成 embedding（EMBEDDED）。
5. 构建混合索引：BM25 + 向量索引（INDEXED/READY）。

### 4.2 在线问答流程
1. 接收 query。
2. 粗检索：章节级混合召回 + 结构先验调整。
3. 细检索：段落级重排（Cross-Encoder 或轻量模型）。
4. 证据融合：去冗余 + 冲突检测 + 置信度加权。
5. 生成答案：调用 LLM（仅通过网关）。
6. 校验答案：反向检索 & 一致性检查。
7. 返回答案 + 证据 + 结构路径 + 可信度。

---

## 5. 推论逻辑（为何可行）

- 层次化检索降低搜索空间 → 提升召回与效率。
- 混合检索结合关键词精确匹配与语义匹配 → 抑制漏召回。
- 证据融合减少冗余与冲突 → 降低幻觉风险。
- 答案校验回查证据 → 提升可解释性与可信度。

---

## 6. 边界条件与反例检查

### 边界条件
- 无明显章节结构：退化为“扁平切分 + 混合检索”。
- 超长段落：启用二次切分与滑窗。
- 多文档冲突：需证据冲突检测与来源优先级策略。
- 低质量 OCR 文档：允许“解析失败 → 仅文本索引”降级。

### 反例与风险
- ⚠️ 反例：检索证据全来自同一章节，导致偏见结论。  
  处理：引入“来源多样性”权重。
- ⚠️ 反例：证据中存在数值冲突。  
  处理：NLI 冲突检测 + 保守输出或标注不确定。
- ⚠️ 反例：LLM 不可用/超时。  
  处理：返回“仅证据摘要”或缓存答案。

## 7. 系统架构（模块划分）

```
┌────────────┐   ┌───────────────┐   ┌───────────┐
│ API Layer  │─▶│ Pipeline Core │─▶│ LLM Gateway│
└────────────┘   └───────────────┘   └───────────┘
        │                 │                 │
        ▼                 ▼                 ▼
┌────────────┐   ┌───────────────┐   ┌───────────┐
│ Doc Store  │   │ Index Service │   │ Verify    │
└────────────┘   └───────────────┘   └───────────┘
        │                 │
        ▼                 ▼
┌────────────┐   ┌───────────────┐
│ Parser     │   │ Retriever     │
└────────────┘   └───────────────┘
```

### 模块清单
- API Layer：REST/JSON，统一请求入口。
- Parser：结构解析（章/节/段落/表格）。
- Chunker：层次化切分，生成路径标签。
- Index Service：BM25 + 向量索引，可插拔。
- Retriever：粗召回 + 精排。
- Evidence Fusion：去冗余、冲突检测、置信度加权。
- LLM Gateway：LLM 生成接口占位。
- Verify：反向检索与一致性校验。
- Feedback：RLHF 反馈入库。

## 8. 数据模型（核心字段）

### 文档 Document
- `id` / `title` / `source` / `status`
- `meta`（作者、时间、类型、语言）

### 结构节点 Section
- `id` / `doc_id` / `level` / `title` / `path`

### 内容块 Chunk
- `id` / `doc_id` / `section_id` / `path`
- `text` / `token_len` / `embedding_id`

### 索引 IndexMeta
- `index_id` / `type`（bm25/vector）
- `status` / `version` / `build_time`

### 证据 Evidence
- `id` / `chunk_id` / `score` / `source_rank`
- `conflict_flag` / `redundant_flag`

### 查询 Query
- `id` / `text` / `created_at`
- `coarse_hits` / `fine_hits`

### 答案 Answer
- `id` / `query_id` / `text`
- `citations` / `confidence` / `verify_status`

### 反馈 Feedback
- `id` / `node_id` / `score` / `comment`

## 9. API 设计（后端接口占位）

### 9.1 文档与索引
- `POST /api/documents/upload`：上传文档
- `GET /api/documents`：列表
- `GET /api/documents/{id}`：详情
- `POST /api/index/build`：触发索引构建
- `GET /api/index/status`：索引状态

### 9.2 查询与问答
- `POST /api/qa/query`
  - 入参：`query`、`top_k`、`rerank_k`
  - 出参：`answer`、`evidence[]`、`citations[]`、`graph`

### 9.3 结构树与证据
- `GET /api/documents/{id}/tree`：结构树
- `GET /api/answers/{id}/evidence`：证据链
- `GET /api/answers/{id}/graph`：证据图谱

### 9.4 LLM 配置（占位）
- `POST /api/llm/config`：保存 LLM 连接配置（不含明文密钥）
- `POST /api/llm/test`：测试连通性

### 9.5 反馈
- `POST /api/feedback`：提交评分/评论

## 11. 算法与评分策略（可验证）

### 11.1 混合检索打分
```
score = α * dense_sim + β * sparse_sim + γ * structure_prior
```
- `structure_prior` 由章节位置/标题相似度/层级深度组成。

### 11.2 证据置信度
```
conf = w1 * match + w2 * consistency + w3 * diversity
```

## 12. 测试与验收基线（Benchmark）

### 功能验收
- 文档解析成功率 ≥ 95%
- 索引构建成功率 = 100%
- 结构树可追溯（chunk 路径准确率 ≥ 90%）

### 检索与融合
- 粗检索召回@N ≥ 0.85
- 精排 NDCG@K ≥ 0.60
- 证据冗余率 ≤ 15%
- 证据冲突率 ≤ 5%

### 生成与校验
- 引用覆盖率（答案句子有证据支撑）≥ 90%
- 答案一致性校验通过率 ≥ 85%

### 性能
- 索引构建：1万 chunk ≤ 5 分钟
- 端到端问答 P95 ≤ 8 秒（外部 LLM 不可控时标注）

## 15. 开发阶段细化（P0-P9，对应 phase-status.csv）

### P0 方案与需求
- 目标：范围冻结与评测基线定义，补齐评测集与标注规范。
- 产出：评审记录、标注模板、至少 50 条问答与证据样本。
- 验收：覆盖事实/流程/比较/数值等问题类型，抽检通过。

### P1 基础设施
- 目标：API 骨架、配置加载、日志追踪、任务队列可用。
- 产出：`/health` 与 `/ready`、结构化日志含 request_id、异步任务可重试。
- 验收：服务可启动、配置校验通过、关键链路可追踪。

### P2 文档处理
- 目标：上传、解析、切分、结构树输出与降级策略落地。
- 产出：`doc_id`、结构树、带路径的 chunk 清单与统计。
- 验收：解析成功率 ≥ 95%，路径准确率 ≥ 90%，解析失败可降级。

### P3 索引构建
- 目标：BM25 + 向量索引构建，支持版本与回滚。
- 产出：`index_id`、`version`、`build_time` 与构建日志。
- 验收：TopN 召回 ≥ 0.85，向量召回 ≥ 0.80，回滚可用。

### P4 检索与重排
- 目标：章节级粗召回 + 结构先验，段落级精排。
- 产出：粗召回候选与精排 TopK 证据候选。
- 验收：召回@N ≥ 0.85，NDCG@K ≥ 0.60。

### P5 证据融合
- 目标：去冗余、冲突检测、置信度与多样性加权。
- 产出：Evidence 列表、冗余/冲突标记与评分。
- 验收：冗余率 ≤ 15%，冲突率 ≤ 5%，证据覆盖率 ≥ 90%。

### P6 生成与校验
- 目标：LLM 网关、证据驱动提示模板、答案校验与回退。
- 产出：`answer`、`citations`、`verify_status` 与回退输出。
- 验收：引用覆盖率 ≥ 90%，校验通过率 ≥ 85%，超时可回退。

### P7 API 接口
- 目标：问答、文档、结构树、反馈、健康检查全量可用。
- 产出：端到端 `query -> answer + evidence + graph` 能力。
- 验收：接口稳定返回，结构图谱可用于前端可视化。

### P8 评测与验收
- 目标：评测集、指标脚本、性能测试与报告产出。
- 产出：MRR/NDCG/覆盖率报告与性能数据。
- 验收：单次评测 < 60s，P95 ≤ 8s（LLM 外部因素单独标注）。

- 产出：敏感信息不落盘，备份恢复流程与演练记录。

## 16. 模块级任务清单（输入/输出/校验）

### 文档上传
- 输入：原始文件与元数据。
- 输出：Document 记录与 RAW 状态，返回 `doc_id`。
- 校验：格式、大小、编码与重复文件检测。

### 结构解析
- 输入：RAW 文档。
- 输出：Section 树与段落列表，状态变更为 PARSED。
- 降级：解析失败转为纯文本索引。

### 层次切分
- 输入：Section 树与段落列表。
- 输出：带路径的 chunk、`token_len` 与统计信息。
- 边界：超长段落二次切分与滑窗。

### 向量化与索引
- 输入：chunk 文本与索引配置。
- 输出：向量库与 BM25 索引，状态变更为 INDEXED/READY。
- 校验：索引可回滚与版本可追溯。

### 粗检索
- 输入：用户 query。
- 输出：章节级候选集合与混合检索分数。
- 规则：结构先验参与评分并可调权重。

### 精排
- 输入：候选章节内的段落集合。
- 输出：TopK 段落级证据候选。
- 规则：可选 Cross-Encoder 或轻量模型重排。

### 证据融合
- 输入：候选证据列表。
- 输出：Evidence 列表、冗余/冲突标记与置信度分数。
- 规则：多样性约束避免同章证据偏置。

### 答案生成
- 输入：问题与证据链。
- 输出：答案文本与引用信息。
- 规则：提示模板约束“基于证据回答”。

### 答案校验
- 输入：答案文本与证据链。
- 输出：`verify_status` 与一致性判定。
- 回退：校验失败或 LLM 超时返回证据摘要。

### 反馈
- 输入：`node_id`、`score`、`comment`。
- 输出：Feedback 记录与可追踪统计。

## 17. 评测与测试落地

### 评测集与标注规范
- 题型覆盖：事实、流程、比较、数值与定义类问题。
- 证据标注：每条答案标注 1-3 个关键证据 chunk。
- 抽检机制：人工抽检与交叉复核。

### 指标与脚本
- 检索指标：Recall@N、MRR、NDCG。
- 融合指标：冗余率、冲突率、证据覆盖率。
- 生成指标：引用覆盖率、一致性通过率。
- 运行约束：单次评测运行 < 60s，输出报告可追溯。

### 性能测试
- 索引构建：1 万 chunk ≤ 5 分钟。
- 在线问答：P95 ≤ 8 秒，外部 LLM 耗时单独标注。
- 监控记录：关键阶段耗时与失败原因归档。

## 18. 风险控制与未决问题

### 风险控制
- 结构缺失：退化为扁平切分 + 混合检索。
- 多文档冲突：冲突检测 + 来源优先级策略。
- LLM 不可用：超时回退至“仅证据摘要”。
- 证据过长：证据裁剪与 TopK 控制。

### 未决问题
- 解析器选择：PDF/DOCX/表格处理库与规则策略。
- 模型选择：Embedding 与重排模型的成本与可替换性。
- 存储与索引：向量库、BM25 引擎与对象存储方案。
- 任务队列：Celery/RQ 与重试策略的最终确定。
